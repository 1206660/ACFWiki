    NEW V4 - Attributes in ACF

    Overview

    Prerequisites: https://slimwiki.com/dark-tower-int/acfu/gameplay-attributes-smw1gncjor-bpzm9lav9vkt

    In ACF, attributes (Health, Stamina, Strength, etc.) are powered by Unreal’s Gameplay Ability System (GAS) — but heavily extended and simplified for designers.

    All the complexity of AttributeSets, GameplayEffects, and custom calculations is wrapped inside a single component:

    UACFGASStatisticsComponent

    This component is the core of ACF’s attribute framework.
    It handles everything from initialization to runtime modification, including:

    • Loading attributes from DataTables or Curves

    • Applying permanent or temporary GameplayEffects

    • Managing level scaling and perk assignment

    • Dynamically creating GameplayEffects at runtime for instant stat changes (damage, healing, buffs, debuffs)

    No manual AttributeSet setup, no static GameplayEffects needed — it’s all automatic.

    Initialization Phase (Base Setup)

    When the game starts, UACFGASStatisticsComponent automatically initializes all your character’s base attributes.

    You can configure the component to:

    • Load attributes from a DataTable (ACFAttributeInits)

    • Or generate them from a CurveTable, scaling with the character’s level

    Blueprint Setup Example

    Event BeginPlay
→ Initialize Attribute Set
→ Apply Permanent Effects

    You can also enable Auto Initialize so this happens automatically on BeginPlay (server side).

    ⚠️ CRITICAL SETUP STEP – READ THIS FIRST ⚠️

    Before using the UACFGASStatisticsComponent,
    you must properly configure the ACF Ability System Component on your character.

    Each AttributeSet that you want to use in ACF (e.g. ACFPrimaryAttributeSet, ACFStatiscsSet, etc.)
    needs to be registered inside your ACF Ability System Component along with its Default DataTable that defines its initial values.

    1. Attribute Initialization from DataTable

    If your character uses static base stats, link a DataTable (ACFAttributeInits) in the component and set a CharacterRow.

    Each row defines a list of Attributes and their initial values.

    At runtime, the component reads that row and applies all values to the character’s Ability System Component (ASC).

    2. Attribute Initialization from CurveTable

    If your character uses level-based scaling, you can instead assign a UCurveTable.

    Each curve defines how an attribute grows per level (e.g., Health, Strength, Stamina).
    When the character’s level increases, ACF automatically updates the attribute’s Base Value.

    Example:

    Curve_Health(level 1 = 100 → level 10 = 300)
Curve_Stamina(level 1 = 50  → level 10 = 200)

    3. Permanent Effects

    You can define an array of Starting Effects in the component.
    Each effect is a GameplayEffect asset applied automatically at initialization — ideal for passive bonuses or class perks (e.g., +10% Defense, +15% Speed).

    Runtime Attribute Management

    Once initialized, UACFGASStatisticsComponent becomes your central runtime controller for all attribute modifications — from damage application to temporary buffs.

    This is where ACF’s GAS integration truly shines: every modification you apply in Blueprint is automatically converted into a GameplayEffect, handled and replicated by GAS under the hood.

    Dynamic Gameplay Effects (Runtime Buffs, Debuffs, and Modifiers)

    At runtime, ACF can generate GameplayEffects dynamically — you don’t need to pre-create assets for each action.

    This is done through FAttributesSetModifier, a data structure that defines what to modify and how.

    AddAttributeSetModifier(const FAttributesSetModifier& Modifier)

    When you call this (in C++ or Blueprint), the component:

    1. Builds a GameplayEffect in memory (using ACF’s internal template).

    2. Fills it with all the modifiers (Attributes, Primary Attributes, and Statistics).

    3. Applies it to the character’s Ability System Component.

    You can even choose between:

    • Permanent modifiers (until manually removed)

    • Timed modifiers (that automatically expire after a duration)

    SetByCaller Gameplay Effects (Runtime Attribute Changes)

    ACF supports fully dynamic, runtime attribute modifications via SetByCaller GameplayEffects.
    This is used by the Blueprint-exposed function:

    ModifyStatistic(FGameplayTag Stat, float Value)

    For example:

    • To apply damage: ModifyStatistic(Health, -50)

    • To heal: ModifyStatistic(Health, +30)

    • To drain stamina: ModifyStatistic(Stamina, -20)

    When called, ACF will:

    1. Create a temporary GameplayEffect based on a configurable template (StatModifierGE).

    2. Inject the value you passed into the GE as a SetByCaller parameter (so no hardcoded values are needed).

    3. Apply it to the ASC immediately.

    4. Let GAS handle execution, replication, and prediction automatically.

    Timed Modifiers

    If you need a temporary buff/debuff (e.g., +20% Attack for 10 seconds), call:

    AddTimedAttributeSetModifier(Modifier, Duration)

    ACF will automatically spawn a GameplayEffect configured with the given duration, and GAS will remove it once expired.

    Removing Modifiers

    Every modifier added returns a FActiveGameplayEffectHandle.
    You can remove it at any time using:

    RemoveAttributeSetModifier(Handle)

    This cancels the effect, instantly reverting the attribute to its previous state.

    Attribute Access and Queries

    The component exposes several Blueprint-accessible functions to easily retrieve current or max values for any stat, using Gameplay Tags as identifiers.

    Function

    Description

    GetCurrentValueForStatitstic(Tag)

    Returns the current (runtime) value of an attribute.

    GetMaxValueForStatitstic(Tag)

    Returns the max (base) value of the same attribute.

    CheckCost(Cost)

    Checks if the character has enough of a stat to perform an action (e.g., Stamina >= 20).

    ConsumeStatistics(Costs)

    Consumes the specified stats using dynamic SetByCaller GameplayEffects.

    Death and Revive Handling

    ACF automatically monitors the default Health attribute.
    When it reaches zero, the component triggers the following:

    • OnDeath_Implementation() (C++ or Blueprint override)

    • OnHealthReachesZero (BlueprintAssignable event)

    When the character is revived:

    • OnRevive_Implementation() is called, restoring attributes to functional values.

    This allows you to implement full death and respawn logic directly in Blueprint, with no manual GAS setup required.
